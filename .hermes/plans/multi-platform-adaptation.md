### 🗿 第一阶段：移动端 Web 深度适配（工期预估：5-7 天）

**核心理念**：斗拱筑基 — 不改变现有架构，在 Tailwind 响应式体系内做增量改进

#### 1.1 设计令牌系统升级

**当前问题**：以 `max-w-4xl` 为硬限制，小屏留白不足

```
现有布局容器:
  <main className="flex-1 max-w-4xl mx-auto w-full px-4 py-6">
  
改造方案:
  <main className="flex-1 w-full mx-auto px-4 sm:px-6 lg:px-8 py-4 sm:py-6 
               max-w-[min(100%,48rem)]">
```

在 `globals.css` 中补充移动端安全区：

```css
/* 新增：安全区与断点优化 */
:root {
  --safe-bottom: env(safe-area-inset-bottom, 0px);
  --safe-top: env(safe-area-inset-top, 0px);
  --page-padding: clamp(1rem, 5vw, 2rem);
}
```

#### 1.2 核心页面改造清单

**🔥 高优先级 — Messages.tsx（私信）**
> *现状：桌面双栏（左侧列表 + 右侧对话），移动端完全不可用*

| 改造项 | 方案 | 工作量 |
|--------|------|--------|
| 移动端单栏切换 | 无 `id` 参数时显示对话列表，有 `id` 时显示对话详情 | 2h |
| 返回按钮 | 移动端对话详情顶部增加 ← 返回列表 | 0.5h |
| 输入框固定底部 | 发送框 `sticky bottom-0` + safe-area | 0.5h |
| 触摸友好 | 对话项增大 paddding，最小高度 56px | 0.3h |

**🔥 高优先级 — Admin.tsx（管理后台）**
> *现状：7 个 Tab 大量数据表格，移动端完全溢出*

| 改造项 | 方案 | 工作量 |
|--------|------|--------|
| Tab 导航横向滚动 | 移动端 `overflow-x-auto` + 激活态指示器 | 1h |

| 操作按钮折叠 | 编辑/删除等操作放入 `...` 更多菜单 | 1h |
| 数据统计卡片 | 4 列 → 2 列 → 1 列渐变 | 0.5h |
| 筛选/搜索弹窗 | 移动端用 `fixed` 底部抽屉 | 1.5h |

**🟡 中优先级 — PostDetail.tsx（帖子详情）**
> *现状：内容相对完善，但评论区操作区在移动端拥挤*

| 改造项 | 方案 | 工作量 |
|--------|------|--------|
| 操作栏弹性换行 | `flex-wrap` + 图标缩小为 `w-4 h-4` | 0.5h |
| 评论区缩进优化 | 嵌套评论移动端最大 `ml-2` 限制 | 0.5h |
| 评论输入框固定 | 大屏原位，小屏 `sticky bottom-0` + safe-area | 1h |
| 大图查看体验 | 图片可点开全屏预览（轻量 viewer） | 2h |

**🟡 中优先级 — Layout.tsx（全局布局）**
> *现状：已有移动端 hamburger，但底部导航缺失*

| 改造项 | 方案 | 工作量 |
|--------|------|--------|
| 底部 Tab 导航 | 手机端添加 `<BottomNav />` 替代顶部菜单 | 3h |
| 搜索栏位置 | 移动端搜索放大镜 → 全屏搜索页 | 1h |
| Header 压缩 | 移动端 `h-12` + 字号缩小 | 0.5h |
| 安全区适配 | 底部导航 + `pb-[env(safe-area-inset-bottom)]` | 0.5h |

**🟢 一般优化 — Home.tsx + Board.tsx + 其他列表页**

| 改造项 | 方案 | 工作量 |

### 🗿 第三阶段：微信小程序适配（工期预估：7-10 天 *可选*）

**核心理念**：大鹏展翼 — 以 Taro 3 为桥，从 React 代码跨端编译至微信小程序

#### 3.1 技术选型对比

| 方案 | 代码复用率 | 学习成本 | 维护成本 | 推荐 |
|------|-----------|---------|---------|------|
| **Taro 3** (React 适配) | 60-70% | 低 | 中 | ✅ |
| 原生微信小程序 | 0% | 中 | 高 | ❌ |
| uni-app | 50% | 中 | 中 | ❌ |
| 单独开发 WebView 壳 | 90% | 低 | 低 | ⚠️ 体验差 |

#### 3.2 Taro 迁移策略

```
campfor-forum/
├── packages/
│   ├── client/          ← 现有 React 18 代码 (保留)
│   └── mini-app/        ← 新增：Taro 小程序项目 (新建)
│       ├── src/
│       │   ├── pages/   ← Taro 页面 (从 client 适配)
│       │   ├── components/ ← 共享组件 (Taro 兼容重写)
│       │   └── stores/  ← 状态管理 (zustand 兼容)
│       └── ...
└── shared/              ← 新增：跨端共享逻辑 (可选)
    └── types/           ← 公共类型定义
```

**迁移清单**：
- 路由方案：React Router → Taro 文件路由
- 组件：DOM 原生标签 → Taro 组件 (`<a>` → `<Navigator>`, `<img>` → `<Image>`)
- API：Axios → Taro.request / 封装统一请求层
- 状态管理：Zustand 可保留（Taro 支持）
- 样式：Tailwind 需权衡（Taro 不原生支持，可用 `weapp-tailwindcss` 插件）

## ⚖️ 第五考据：典故溯源与技术验证

| 概念 | 传统意象 | 技术映射 |
|------|---------|---------|
| **榫卯** | 木构件的凹凸咬合，无缝衔接 | 组件间渲染层 + 逻辑层的解耦适配 |
| **斗拱** | 从柱头层层出挑承重 | 渐进式适配：基础层 → 增强层 → 平台层 |
| **留白** | 计白当黑，疏可走马 | 移动端通过 `clamp()` 函数弹性控制间距，宁可缩小不可溢出 |
| **水墨渐变** | 由浓转淡的自然过渡 | Tailwind 响应式前缀：`sm:` → `md:` → `lg:` 的语义化断点 |
| **界画** | 以界尺引线，工整精确 | Admin 表格移动端转卡片，遵循 iOS HIG + Material Design 双规范 |

> **天权收笔：**
> 此规划以"斗拱三叠"为骨架 — 第一叠以 Tailwind 响应式前缀为基、聚焦移动端 Web；第二叠以 PWA 为肉、打磨离线与推送体验；第三叠以 Taro 为翼、拓展微信生态。先固本（Phase 1），再养气（Phase 2），最后展翅（Phase 3）。
>
> 不提倡大改架构，而是"削木为楔，因材施榫"——在现有 Tailwind 体系内用最小成本做最大覆盖。
>
> 📜 *榫卯合则千钧不坠，斗拱叠则飞檐凌云。*
