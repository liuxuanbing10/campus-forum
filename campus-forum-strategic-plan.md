**战略隐喻：治大国若烹小鲜**

| 阶段 | 时间 | 核心主题 | 传统文化隐喻 |

## 第一阶段 · 固本（第 1-4 周）— 质量基础设施

> **为什么做：** 当前项目测试覆盖率极低、类型安全薄弱、没有任何缓存。这些是技术债的"本金"，不还则利滚利，后续每行新代码都在积累风险。相当于**修城墙前先固地基**。

### P0-1：单元测试体系建设

| 维度 | 内容 |
|------|------|
| **为什么做** | 当前仅 3 个测试文件（websocket.test.ts、MetaManager.test.ts、api.test.ts），且 api.test.ts 几乎为空。无测试 ≈ 无安全网，每次重构都像走钢丝。 |
| **做什么** | 建立覆盖核心层（core）、数据库层（database）、各插件的单元测试。 |
| **怎么做** | ① 为 `core` 包编写 PluginManager、EventBus 单元测试（6-8 个用例） ② 为 `database` 包编写 schema 初始化、migration 幂等性测试（4-6 个用例） ③ 为每个插件编写核心 handler 的测试（每个插件至少 5 个用例） ④ 使用 vitest + 内存 SQLite 实现无外部依赖测试 |
| **交付物** | 测试覆盖率达 40%+，CI 流程集成 `vitest run` |
| **排期** | 第 1-2 周 |
| **战略隐喻** | **"曲突徙薪"** — 《汉书·霍光传》：与其事后救火，不如提前预防。 |

### P0-2：TypeScript Strict 模式 + 类型清理

| 维度 | 内容 |
|------|------|
| **为什么做** | 代码中大量使用 `any`（特别是 admin 插件和 server 入口），类型定义存在不一致（如 `avatar_url` vs `avatarUrl`、`isBanned` vs `is_banned` 混用），接口定义与实际返回不匹配。 |
| **做什么** | 启用 `strict: true`，清理所有 `any`，统一命名规范（snake_case / camelCase 映射）。 |
| **怎么做** | ① `tsconfig.json` 设置 `strict: true` + `noImplicitAny: true` ② 逐文件消除类型错误（从 core → database → server → plugins 自上而下） ③ 建立 API 响应类型与前端类型的一致性契约 ④ 引入 zod 运行时校验与 TypeScript 类型联动（`z.infer`） |
| **交付物** | `strict: true` 编译通过，`any` 使用量减少 90%+ |
| **排期** | 第 2-3 周（与 P0-1 并行） |
| **战略隐喻** | **"不以规矩，不能成方圆"** — 孟子《离娄章句上》 |

### P0-3：缓存层引入

| 维度 | 内容 |
|------|------|
| **为什么做** | 当前所有请求直击 SQLite，无任何缓存。随着用户增长，热门帖子/板块频繁读取会造成性能瓶颈。SQLite 单写特性在高并发下更为明显。 |
| **做什么** | 引入内存缓存（开发/小规模）或 Redis（生产）缓存层。 |
| **怎么做** | ① 在 `core` 包中抽象 `CacheAdapter` 接口（get/set/del/clear） ② 先用 `node-cache` 或 `lru-cache` 做内存实现 ③ 缓存热门帖子列表（TTL 60s）、板块信息（TTL 300s）、用户信息（TTL 120s） ④ 通过 EventBus 发布缓存失效事件（帖子更新时主动清除缓存） ⑤ 后续可无缝切换至 Redis 实现 |
| **交付物** | 缓存适配器 + LRU 实现 + 核心 API 的缓存策略 |
| **排期** | 第 3-4 周 |
| **战略隐喻** | **"蓄水养鱼"** — 先建水库、再放鱼苗，而非临渴掘井。 |

### P0-4：前端组件测试 + 小型 E2E

| 维度 | 内容 |
|------|------|
| **为什么做** | 前端目前只有 1 个组件测试（MetaManager），其余页面完全无防护。核心交互流程（登录→发帖→评论）需要 E2E 覆盖。 |
| **做什么** | 为核心组件编写 RTL 测试，关键用户流程编写 Playwright E2E。 |
| **怎么做** | ① 使用 Vitest + React Testing Library 测试 Layout、PostListItem、MarkdownEditor 等核心组件 ② 使用 Playwright 编写 3 条 E2E 黄金路径（登录→浏览→发帖、搜索→阅读→评论、注册→设置→查看通知） ③ 集成到 CI 中 |
| **交付物** | 20+ 组件测试用例 + 3 条 Playwright E2E |
| **排期** | 第 3-4 周（与 P0-3 并行） |
| **战略隐喻** | **"如履薄冰"** — 小心翼翼验证每一个关键节点。 |

## 第二阶段 · 强基（第 5-8 周）— 架构与运维

> **为什么做：** 基础质量打好后，需要构建可生长的架构底座。当前 DevOps 缺失、文件存储不合理、没有国际化能力，这些制约了项目从小型 demo 走向真正可运营的产品。相当于**加固城防、修通驿道**。

### P1-1：Docker 化 + CI/CD 流水线

| 维度 | 内容 |
|------|------|
| **为什么做** | 目前仅有两个 shell 脚本（deploy.sh、keep-alive.sh），无 Docker 镜像，无 CI 自动测试+构建。每次部署都是手工操作，易出错、不可重复。 |
| **做什么** | 编写生产级 Dockerfile + docker-compose + GitHub Actions CI/CD。 |
| **怎么做** | ① 多阶段 Dockerfile（build stage → prod stage，~100MB 镜像） ② docker-compose.yml 包含 server + turso/db 服务 ③ GitHub Actions：push 时运行 lint + test + build，tag 时自动构建镜像 ④ 可选：部署到阿里云 ECR + ECS 自动发布 |
| **交付物** | Dockerfile + docker-compose.yml + GitHub Actions workflow |
| **排期** | 第 5 周 |
| **战略隐喻** | **"兵马未动，粮草先行"** — 部署管线就是军队的粮道。 |

### P1-2：文件存储重构（S3 兼容对象存储）

| 维度 | 内容 |
|------|------|
| **为什么做** | 图片以 base64 存入 SQLite TEXT 字段（`uploaded_images.data`），这是反模式——数据库体积暴增、备份困难、无法 CDN 加速。 |
| **做什么** | 迁移至本地磁盘文件系统 + S3 兼容对象存储（阿里云 OSS / MinIO）。 |
| **怎么做** | ① 在 `plugins/uploads` 中抽象 `StorageAdapter`（本地 fs / S3） ② 本地开发使用 fs 存储到 uploads/ 目录 ③ 生产环境接入阿里云 OSS（或 MinIO 自建） ④ 前端直传签名 URL（避免后端带宽瓶颈） ⑤ 数据库迁移脚本：将现有 base64 数据批量迁移至文件存储 |
| **交付物** | StorageAdapter 接口 + 本地实现 + OSS 实现 + 数据迁移脚本 |
| **排期** | 第 5-6 周 |
| **战略隐喻** | **"移山填海"** — 将压在山下的东西（DB 中的 base64）搬到大江大海（对象存储）中。 |

### P1-3：国际化（i18n）

| 维度 | 内容 |
|------|------|
| **为什么做** | 校园论坛可能吸引国际学生（留学生、交换生）。全中文硬编码严重限制了用户群体。 |
| **做什么** | 引入 react-i18next 实现前端多语言（中/英/日/韩），后端错误消息国际化。 |
| **怎么做** | ① 安装 react-i18next 并配置语言检测（浏览器偏好 + localStorage 覆盖） ② 提取所有中文文案为 JSON 翻译文件（`en.json`、`zh.json`、`ja.json`、`ko.json`） ③ 后端错误消息通过 Accept-Language 头返回对应语言 ④ 管理员后台添加语言切换开关 |
| **交付物** | 4 语言支持 + 语言切换 UI + 后端错误消息国际化 |
| **排期** | 第 6-7 周 |
| **战略隐喻** | **"海纳百川"** — 有容乃大，兼容并包。 |

### P1-4：站外通知推送（邮件 + WebPush）

| 维度 | 内容 |
|------|------|
| **为什么做** | 当前通知仅在站内显示，用户离开网站就收不到任何通知。邮箱验证也是"演示模式"，没有真实邮件发送。 |
| **做什么** | 集成邮件发送（Nodemailer）和浏览器 WebPush 推送。 |
| **怎么做** | ① Nodemailer 接入 SMTP（QQ邮箱 / SendGrid / 阿里云邮件推送） ② 真实邮箱验证流程（发送验证码、验证邮箱） ③ 通知时发送邮件摘要（可配置频率：实时/每日摘要/关闭） ④ Web Push API 推送浏览器通知（需要 service worker + VAPID keys） |
| **交付物** | 邮件发送模块 + 真实邮箱验证 + WebPush 订阅/推送 |
| **排期** | 第 7-8 周 |
| **战略隐喻** | **"烽火传讯"** — 古代烽火台传递军情，今日通知应主动触达。 |

### P1-5：数据库读写分离 + 连接池优化

| 维度 | 内容 |
|------|------|
| **为什么做** | SQLite 单文件单写特性是并发瓶颈。随着用户量增长，需要读写分离或迁移至 PG/MySQL。 |
| **做什么** | 分析 SQLite 瓶颈，引入 WAL 模式，规划数据库迁移路径至 PostgreSQL。 |
| **怎么做** | ① 启用 SQLite WAL 模式（`PRAGMA journal_mode=WAL`）提升并发读性能 ② 抽象 DatabaseAdapter 层，确保所有插件使用 adapter 而非裸 SQL ③ 编写 pg/MySQL 版 DatabaseAdapter 实现 ④ README 中增加数据库迁移指南 |
| **交付物** | WAL 模式启用 + 多数据库适配器 + 迁移文档 |
| **排期** | 第 8 周 |
| **战略隐喻** | **"多路并行"** — 大禹治水，疏而非堵。 |

## 第三阶段 · 拓疆（第 9-12 周）— 创新与增长

> **为什么做：** 地基已稳、城池已固，此时可出击开拓。AI 整合、SEO 优化、数据驱动、体验提升是让 campus-forum 从"能用"到"好用"的质变。相当于**开疆拓土、建文治之功**。

### P2-1：AI 内容推荐 + 智能审核

| 维度 | 内容 |
|------|------|
| **为什么做** | 当前帖子排序仅按时间/热度，缺乏个性化推荐；敏感词过滤依赖硬编码列表，无法识别变体/上下文；审核队列需人工处理。 |
| **做什么** | 引入 AI 驱动的推荐引擎、内容审核助手、自动标签生成。 |
| **怎么做** | ① 用户行为埋点（浏览、点赞、收藏、停留时长）→ 建立用户兴趣向量 ② 基于协同过滤 + 内容标签的混合推荐算法（初期可用简单策略：同板块热门、关注者推荐） ③ 对接 LLM API（DeepSeek / Qwen）做内容审核：自动判断违规内容、生成审核理由 ④ 自动标签生成：基于帖子内容提取关键词 |
| **交付物** | 推荐 API + LLM 审核插件 + 自动标签服务 |
| **排期** | 第 9-10 周 |
| **战略隐喻** | **"如虎添翼"** — 传统论坛加上 AI 算力，恰似猛虎生翼。 |

### P2-2：SEO 深度优化

| 维度 | 内容 |
|------|------|
| **为什么做** | 论坛内容天然适合搜索引擎索引，但当前 SPA 无 SSR、无 OG 标签、无 sitemap.xml，搜索引擎基本抓不到内容。 |
| **做什么** | 实现 SSR/SSG、OG 标签、结构化数据、sitemap 自动生成。 |
| **怎么做** | ① 使用 Vite SSR（或 @vitejs/plugin-ssr）实现帖子详情页的服务端渲染 ② 帖子页添加 OG 标签（标题、描述、图片、作者） ③ 实现 `/sitemap.xml` 动态生成（按板块分页） ④ 添加 `robots.txt` 优化爬虫策略 ⑤ Google Search Console 验证 | 结构化数据（JSON-LD `BreadcrumbList`、`DiscussionForumPosting`） |
| **交付物** | SSR 帖子详情页 + OG 标签 + sitemap.xml + 结构化数据 |
| **排期** | 第 9-10 周（与 P2-1 并行） |
| **战略隐喻** | **"酒香也怕巷子深"** — 内容再好，搜不到也是白费。 |

### P2-3：WebSocket 房间管理 + 实时频道

| 维度 | 内容 |
|------|------|
| **为什么做** | 当前 wsService 是全局广播模式，无房间/频道概念。用户会收到全站通知而非仅关注的内容。 |
| **做什么** | 实现 WebSocket 房间订阅系统，按频道分发消息。 |
| **怎么做** | ① 引入房间管理（`room.join` / `room.leave` / `room.broadcast`） ② 频道划分：post_updates、comments、notifications、team_messages ③ 用户 WS 连接时自动订阅相关频道 ④ 后端通过 EventBus 分发消息到对应房间 |
| **交付物** | WS 房间管理器 + 频道分发 + 前端自动订阅 |
| **排期** | 第 10-11 周 |
| **战略隐喻** | **"各行其道"** — 车走车路、马走马路，消息亦需分流。 |

### P2-4：数据看板增强 + 用户行为分析

| 维度 | 内容 |
|------|------|
| **为什么做** | 当前 Admin 统计看板仅有基础 SQL 聚合，无可视化图表、无趋势预测、无用户行为漏斗分析。运营决策靠"猜"。 |
| **做什么** | 引入 ECharts/Chart.js 可视化 + 用户行为漏斗分析。 |
| **怎么做** | ① 前端集成 ECharts（或 Chart.js + react-chartjs-2） ② 新 API：用户留存率、发帖转化漏斗（浏览→注册→发帖→持续活跃） ③ 7 天/30 天趋势可视化（用户增长、帖子趋势、互动率） ④ 活跃用户分布热力图（按小时+星期） ⑤ 看板导出为 PDF/Excel |
| **交付物** | 可视化看板 + 用户漏斗分析 + 热力图 |
| **排期** | 第 10-11 周 |
| **战略隐喻** | **"运筹帷幄"** — 知己知彼，百战不殆。数据就是情报。 |

### P2-5：移动端深度优化 + PWA 增强

| 维度 | 内容 |
|------|------|
| **为什么做** | 校园用户以移动端为主，当前仅做了 Tailwind 响应式，无独立移动端体验优化、无 PWA 离线支持。 |
| **做什么** | 移动端交互优化、PWA 离线缓存、App Shell 架构。 |
| **怎么做** | ① 移动端底部导航栏（Tab Bar）替代侧边汉堡菜单 ② 帖子列表无限滚动替代分页 ③ 移动端手势支持（下拉刷新、左滑返回） ④ 完善 service worker，实现离线缓存策略（Cache-First for static, Network-First for API） ⑤ manifest.json 增加 splash screen、主题色、图标集 |
| **交付物** | 移动端 Tab 导航 + 无限滚动 + PWA 离线可用 |
| **排期** | 第 11-12 周 |
| **战略隐喻** | **"以柔克刚"** — 移动端如流水，适配其势而非对抗。 |

### P2-6：API 文档 + 架构文档 + 开发者门户

| 维度 | 内容 |
|------|------|
| **为什么做** | 当前仅有 README 和 CHANGELOG。没有 API 文档、没有架构决策记录（ADR）、没有开发指南。阻碍社区贡献和后续维护。 |
| **做什么** | 生成 Swagger/OpenAPI 文档 + 撰写架构文档。 |
| **怎么做** | ① 集成 `@fastify/swagger` 自动生成 OpenAPI 3.0 文档 ② swagger-ui 路由挂载（`/docs`） ③ 撰写 ADR（Architecture Decision Records）：记录每个插件的设计决策 ④ 更新 CONTRIBUTING.md 为完整开发者指南（本地开发环境搭建、插件开发教程、部署流程） ⑤ 可选：GitHub Pages 托管文档站点 |
| **交付物** | Swagger UI + ADR 文档 + 完整开发者指南 |
| **排期** | 第 12 周（收尾总结） |
| **战略隐喻** | **"名正言顺"** — 孔子曰："名不正则言不顺，言不顺则事不成。"文档即正名。 |

## 风险矩阵

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|----------|
| SQLite 在生产环境达到性能瓶颈 | 中 | 高 | 优先启用 WAL 模式，提前规划 PG 迁移路径 |
| AI 服务（LLM）调用成本超出预期 | 中 | 中 | 使用本地小模型（如 Qwen-1.8B）做轻量审核，仅关键内容走云端 |
| 重构引入回归 bug | 高 | 中 | P0-1 测试先行，确保每个重构都有测试覆盖 |
| 开发者动力衰减 | 中 | 高 | 每阶段设置可演示的里程碑，保持成就感 |
